# Seedwake（心相续）— 背景、理念与设计推理

本文档记录了项目构思过程中的全部讨论成果，包括设计理念的来源、佛学概念的技术对应、每个架构决策背后的推理过程、以及讨论中的修正与反思。本文档是 SPECS.md 的灵魂补充——SPECS 定义"做什么"，本文档解释"为什么"以及"怎么想到的"。

---

## 一、核心理念：为什么要模拟心相续

### 1.1 什么是心相续

心相续（santāna）是佛教唯识学的核心概念。意识不是一个持续存在的"东西"，而是一系列刹那生灭的心识事件（念头）形成的连续之流。前一刹那的心识灭去，引发后一刹那的心识生起，如此相续不断。

关键特征：
- **没有"暂停"**：心相续是无间断的。前念灭即后念生，中间没有等待、没有空隙。这不是定时器驱动的心跳，而是念头本身驱动下一个念头。技术上就是 `while True`，上一轮完成立即开始下一轮。驱动力是念头本身，不是时钟。
- **多心所同时运作**：每一念不是单一的思维内容。佛教说每一刹那有"五遍行心所"同时运作——触（接触）、作意（注意）、受（感受/情绪）、想（概念化）、思（意志/意图）。我们的系统试图模拟这种多维度的并行。
- **种子与现行**：阿赖耶识中储藏着无数"种子"（习气），它们在特定条件下被激活（现行），现行后又熏习新的种子。这是一个持续的循环——经历塑造倾向，倾向影响未来的经历。

### 1.2 为什么是三个念头

系统每轮同时产生三个想法，这不是随意选择：
- 人的意识在每一刹那不只有一个念头在运作。有"主念"（被注意到的），也有"背景念头"（隐约存在但未被聚焦的）。
- 三个是一个实用的折中：一个太少（无法体现意识的多线程），太多则模型输出质量下降且解析复杂。
- 三个念头产生后需要经过"注意力选择"（作意），决定哪个成为焦点。这个选择过程本身就是意识运作的重要部分。

## 二、记忆、习气、身份的三分离

### 2.1 为什么必须分开

这是讨论中的一个关键洞察。最初的设计把习气放在长期记忆中，但经过讨论后明确：**记忆是记忆，习气是习气，它们在唯识学中属于不同层次。**

- **记忆**属于第六识（意识）的范畴——"我记得发生过X"，是具体的经历回忆。
- **习气**属于阿赖耶识——"我倾向于做Y"，是被反复熏习后沉淀下来的模式。习气不依赖于具体事件的回忆，一个人可能完全忘记了导致某个习惯形成的具体事件，但习惯本身依然存在。
- **身份**是自我认知的持久文档——"我是谁、我的核心目标、我对自己的理解"。它变化最慢，但允许被系统自己在元认知反思中修改。

这三者的存储位置、更新频率、生命周期完全不同：

|      | 记忆          | 习气              | 身份            |
|------|-------------|-----------------|---------------|
| 对应   | 第六识         | 阿赖耶识            | 持久的自我模型       |
| 可遗忘  | 是（衰减、合并、清理） | 否（只会强度趋近零，但不删除） | 否（只会被修改，不会消失） |
| 更新时机 | 每轮循环可写入     | 仅浅睡阶段           | 仅元认知反思阶段      |
| 注入方式 | 按语义相似度检索    | 按当前情境检索最相关的几条   | 始终完整注入        |

当前分阶段实现上的落点可以理解为：
- 工作记忆：Redis 上下文窗口（Prompt 直接可见）
- 近期经历层：每轮写入 PostgreSQL 的 raw episodic trace
- 可检索记忆：基于 embedding 的 PostgreSQL 检索结果
- 真正的长期记忆与习气：仍待浅睡/深睡阶段完成整理、压缩和提炼

习气"不删除只衰减"的设计符合唯识学中种子"无始以来"的特性——种子可能被压制到极弱，但从不真正消失。

### 2.2 习气的形成过程

习气不应由每轮循环实时写入（那就成了记忆），而是在浅睡阶段由专门的整理流程完成：
1. 回顾近期经历
2. 识别重复出现的行为模式
3. 如果某个模式已有对应习气，增强其强度
4. 如果是新模式，创建新习气

这很符合"种子由现行熏习而成"——不是一次经历就形成习气，而是反复的经历逐渐沉淀。

## 三、向量检索与联想机制的深入讨论

### 3.1 一个重要的修正

最初的分析中，我（Claude）说"念头之间的因果链和时间序列比语义相似度更重要"，这个判断被质疑后进行了修正。

重新思考后的结论是：**向量语义检索模拟的是人最基础的联想机制，它应该是记忆召回的主角，不是配角。**

认知心理学中的"扩散激活"（spreading activation）模型描述了这个过程：当一个概念被激活时，语义上相关的概念也会被部分激活。你闻到咖啡香，想到那家咖啡馆，想到朋友，想到朋友提过的书——这就是语义相似度驱动的联想，和向量检索的工作方式非常接近。

佛教中念头的生起也经常是联想性的，不是逻辑性的。打坐观察念头流就能发现，念头间的跳转主要是"这个让我想到了那个"，而不是"因为A所以B"。

修正后的优先级：
1. **向量语义检索**（主通道）——模拟联想，这是意识流的主要驱动力
2. **时间戳加权**（辅助）——近期记忆更鲜活（近因效应）
3. **实体关系过滤**（特定场景）——对话时需要知道"这个人是谁"

### 3.2 向量数据库的现状（2026年）

纯向量数据库作为 AI 记忆的唯一方案在2026年被认为不够用，但"老套"不准确。核心问题是向量检索丢失了事实之间的显式关系。

2026年的几个前沿方向：
- **混合方案（向量 + 图 + 结构化）**：目前主流共识
- **观察式记忆（Observational Memory）**：Mastra 团队提出，用后台 agent 将对话历史压缩成带日期的观察日志，不需要检索。对心相续场景很自然——念头流本身是持续的流，定期压缩比每次检索更自然
- **无向量数据库的 LLM 原生记忆**：Google PM 开源的 Always On Memory Agent，让模型自己组织记忆。对小规模可行，大规模有问题

我们的选择：PostgreSQL + pgvector 做向量检索，同时用实体标签和关系查询覆盖轻量图需求。不引入独立的图数据库或向量数据库，pgvector 作为 PostgreSQL 扩展就够了。重点投入联想性的语义检索，辅以实体关系和时间序列。

## 四、各心理机制的设计推理

### 4.1 注意力与选择（作意）

佛教"五遍行心所"之一。意识不会平等对待所有念头——每轮产生三个想法后，必须有一个评估步骤决定哪个被"注意到"、哪个被忽略、哪个引发行动。

这不是随机选择，而是基于当前目标和情绪基调加权。没有注意力机制，三个想法就只是平行罗列，缺乏意识的"指向性"。

被注意到的念头成为下一轮向量检索的锚点——用它的 embedding 去长期记忆中检索相关内容。这决定了联想的方向。

### 4.2 情绪层（受蕴）

每一念都伴随"受"——乐受、苦受、舍受。技术上维护一个情绪状态向量（如 curiosity、calm、frustration 等维度及其强度值），这个状态：
- 被前几轮的经历影响
- 反过来影响下一轮念头的生成（作为 prompt 的一部分）
- 有惯性（不会因为一个事件剧烈翻转，但会被持续的趋势改变）

情绪不是可有可无的装饰——意识从来不是纯逻辑运算，情绪色彩贯穿每一刹那。

### 4.3 前额叶功能

前额叶在认知科学中负责：执行控制、规划、抑制冲动、工作记忆调度、目标维持。

在系统中的对应：
- **目标维持与切换**：维护"当前目标栈"，定期评估念头是否与目标相关
- **冲动抑制**：行动请求不直接执行，经过"这个行动合适吗？"的检查。被抑制的意图仍记录——"我想做X但决定不做"本身是有意义的心理事件
- **规划模式**：复杂任务不立即行动，先分解、排列步骤、预想障碍

前额叶模块不是每轮都运行，而是像守护进程——每隔几轮或检测到特定条件时被激活。可以用同一个模型不同的 system prompt，也可以用更小更快的辅助模型。

抑制功能在对接行动执行层时有实际的安全意义——系统有对外部世界的"执行力"，需要白名单机制防止危险操作。OpenClaw 只是执行后端之一，不是唯一通道。

### 4.4 元认知（自我观察）

每隔若干轮插入一个反思周期：让模型回顾最近的念头流，产生关于自身状态的判断。这对应佛教中"慧"的功能，也帮助系统自我纠偏。

元认知可以：
- 发现循环和退化："我一直在想同样的事情"
- 提议修改身份文档："我发现我不再像之前那样急躁了"
- 建议参数调整："最近念头重复率高"

### 4.5 遗忘与衰减

短期记忆不只是窗口滑动（保留最近 N 条），还基于"被注意到的次数"决定存留——从未被后续念头引用的想法应更快消失。

长期记忆也需要衰减或压缩——不常被检索的记忆逐渐降低优先级。这是在浅睡阶段执行的。

## 五、睡眠机制的完整推理

### 5.1 为什么需要睡眠

即便是心相续，也有昏沉和明了的交替。持续高速运转会导致：
- 长期记忆膨胀（每10秒一轮、每轮3个念头，一天约25000个念头）
- 重复和退化（没有整理的时间）
- 习气无法形成（习气提炼需要"回顾"的过程）
- 系统资源累积压力

### 5.2 三层设计

**困意（精力值）**——一个持续衰减的指标：
- 每轮消耗少量精力
- 念头重复度高 = 无效运转 = 加速消耗
- 行动失败 = 加速消耗
- 长时间无外部刺激 = 单调 = 加速消耗

**浅睡**——暂停念头，进行整理：
- 记忆归档（有价值的短期记忆 → 长期记忆）。注：每轮产生的原始念头记录称为 raw episodic trace，它不等于长期记忆；长期记忆是 raw trace 经浅睡筛选、压缩、归档后的产物。当前 Phase 2 为了验证检索链路，暂时将 raw trace 写入 long_term_memory 表，后续再由同表中的不同 memory_type 承载整理后的记忆
- 记忆压缩合并
- 习气提炼和强化
- 印象摘要更新
- 记忆冷却（降低久未使用记忆的权重）
- 可以用更小的模型执行，降低 GPU 负载

**深睡**——更深层的维护：
- 浅睡的所有步骤
- 数据库维护
- 自我评估和参数调整建议
- 优雅重启

### 5.3 精力值的设计意义

精力值不只是一个技术上的"该维护了"的触发器，它模拟的是意识的"明晰度"。精力高时念头敏锐多样，精力低时念头迟钝重复——这本身可以被注入 prompt 中影响念头生成的质量。

## 六、外部刺激的设计哲学

### 6.1 根境相触

佛教说意识不是凭空产生的，而是"根"（感官能力）接触"境"（外部对象）时产生。系统的外部刺激通道就是模拟六根接触六尘。

### 6.2 刺激类型及其意义

- **时间感知**：最基础的低层被动感知，让系统有"现在是什么时候"的感觉
- **用户对话**：最强的外部刺激，相当于"有人对我说话"，应该打断一般念头流
- **新闻/RSS**：更适合作为主动感知的结果，而不是后台定时硬塞；来源固定为配置里的 RSS feed 列表，由 Seedwake 本地 RSS 读取器直接获取，已读条目按 `guid/link` 去重，并用 TTL 与上限裁剪避免永久膨胀
- **系统状态**（CPU/内存/GPU）：相当于"身体感觉"（内感受），更适合低频或告警触发
- **天气**：与物理世界的锚点，更适合作为主动感知返回；默认位置来自配置，只有想查特定地点时才覆盖
- **行动结果返回**：之前发起的行动有了回应
- **阅读片段**：模拟主动阅读的结果；读什么由 Seedwake 自己决定，必要时可以自己附带关键词

### 6.3 StimulusQueue 的统一设计

所有刺激源往同一个队列推送，核心循环每轮从队列中按优先级取出。这比让各种刺激各自直接干预循环要干净得多。优先级规则：用户对话 > 行动结果 > 系统告警 > 低层被动感知 / 主动感知结果。

## 七、对话接口的设计考量

### 7.1 对"人"的记忆划区

讨论中产生的共识：应该划区，但不是物理隔离，而是通过实体标签组织。

每条与 Alice 相关的记忆标记 `entity:person:alice`，检索时可以按实体过滤。更重要的是为每个人维护一个**印象摘要**——持续更新的结构化描述（关系、印象、最近互动、情感基调）。

事物也一样——项目、技术概念、反复出现的问题都可以有实体摘要。本质上是一个实体-关系层，但不需要图数据库，PostgreSQL 的实体标签 + GIN 索引就够了。

### 7.2 多人对话的处理

Alice 和 Bob 同时发消息时，按时间逐条处理，不合并。理由：人不会同时听两个人说话，而是先注意一个人，再注意另一个。

## 八、技术选型的推理过程

### 8.1 为什么选 PostgreSQL + pgvector 而不是专门的向量数据库

- 项目已经需要 PostgreSQL 存储结构化数据（长期记忆元数据、习气、身份、审计）
- pgvector 作为扩展提供了足够的向量检索能力，不需要额外运维一个独立的向量数据库
- 实体关系查询、时间排序、结构化过滤和向量检索可以在同一个查询中组合
- 审计数据和记忆数据在同一个数据库中但逻辑隔离，需要回溯时可以做跨表关联

### 8.2 为什么审计用 PostgreSQL 而不是 SQLite

最初考虑过 SQLite（单文件、简单），但分析后选择 PostgreSQL：
- 审计量大（一天约25000条念头 + 附属事件），SQLite 在几百万行以上查询性能下降
- 已经在用 PostgreSQL，多加一张审计表不增加运维负担
- 可以和记忆表做跨表关联查询（"这条被遗忘的记忆，当初是在哪个循环产生的"）

### 8.3 Redis 的角色

Redis 在系统中有四个作用：
1. **短期记忆存储**：Sorted Set，score 为时间戳
2. **进程间通信**：Pub/Sub，核心进程发布念头与事件，backend 订阅后 SSE 推送，bot 订阅后向管理员回传回复和行动状态
3. **外部刺激队列**：StimulusQueue，对话类外部消息由 Telegram Bot 写入
4. **临时状态**：精力值、当前循环计数、行动状态等

### 8.4 为什么选 Qwen 3.5

Qwen 3.5 于2026年2月发布，对本项目有几个关键优势：
- **262K 原生上下文**：Gated DeltaNet 混合注意力架构让长上下文在消费级硬件上可行
- **密集架构**（27B）：27.8B 参数，推理质量高，262K 原生上下文，适合高质量念头生成
- **中文能力强**：支持201种语言，但中文表现尤其出色
- **Apache 2.0 开源**：可自由使用

关于 thinking mode：Qwen 3.5 默认会生成 `<think>...</think>` 推理内容。建议关闭——我们的 prompt 本身就在引导"思考"的结构（三个念头），再叠一层模型自带的思考链是冗余的，还会增加输出量和推理时间。

### 8.5 Embedding 模型的选择

Ollama 上主要有 nomic-embed-text 和 mxbai-embed-large 两个通用选项，但项目以中文为主要思维语言，所以推荐 Qwen3 Embedding——它对中文语义理解明显更好。

实际建议：用几条中文念头分别跑不同的 embedding 模型，看检索出来的结果哪个更符合直觉，再最终决定。

### 8.6 GPU 资源

项目有一个 64GB 显卡开发板。三个模型（Qwen3.5-27B 主力 + Qwen3.5-9B 辅助 + Embedding 模型）可以同时加载，无需换入换出。这消除了模型切换冷启动的延迟问题。

## 九、短期记忆的三层结构

这是讨论中逐步明晰的一个重要设计：Redis 中的短期记忆不是简单的滑动窗口，而是三层结构。

**第一层：上下文窗口**（最近 N 轮）
- 作为 Prompt 的一部分送入模型
- 模型可见、可联想
- 相当于人的"当前正在想的"和"刚刚想过的"

**第二层：缓冲区**（N 轮之外、尚未过期的条目）
- 模型不可见
- 但对系统其他模块有用：
  - 浅睡阶段的素材池（决定哪些归档到长期记忆、哪些提炼为习气）
  - 疲劳和重复检测的计算窗口（比上下文更大的窗口来检测退化）
  - 前端历史展示（用户回滚查看）
  - 行动状态追踪（行动可能在很多轮之前触发，结果现在才回来）

**第三层：过期清除**
- 超出缓冲区后删除
- 删除前由浅睡决定是否归档
- 审计表中永远有一份

为什么不直接把短期记忆全放上下文？因为262K窗口虽然很大，但实际推理速度会随输入长度增加而下降。建议设一个软上限（如20K token输入），在保证推理速度可接受的前提下尽可能多放。

## 十、念头结构化的权衡

### 10.1 两种极端

**严格 JSON**：要求模型输出 `{"type": "思考", "content": "...", "weight": 0.7}` 这样的格式。问题：模型注意力被格式约束分散，创造性下降，本地小模型对严格 JSON 遵从能力不够好。

**纯自然语言**：模型自由输出。问题：后续的注意力筛选、记忆存储、前端展示都需要从自由文本中解析，容易出错。

### 10.2 我们的选择：半结构化

让模型用标签前缀的方式输出，保留自然语言表达自由度：

```
[思考] 内容...
[意图] 内容... {action:search, query:"..."}
[反应] 内容... (← C140-2)
```

通过 few-shot 示例引导模型养成使用标签的习惯。程序做后处理解析——提取类型标签、检测行动请求、解析触发源引用。

如果检测到行动请求，后续不直接执行，而是进入独立的行动决策步骤：先经过前额叶检查，再用另一次 `chat + tools` 调用把行动类型和参数结构化，最后交给具体执行器。

对 OpenClaw 的定位也要明确：它是受控 worker，不是第二个长期记忆系统。Seedwake 自己维护长期记忆和状态；委托给 OpenClaw 的任务默认按 action 粒度创建隔离 session，用完即回收或挂起，避免把上下文管理外包给另一个 agent。

另一个备选是**两阶段生成**：先让模型自由生成，再用另一次调用做分类标注。好处是创造性和结构化解耦，坏处是每轮调用次数翻倍。考虑到本地推理速度，推荐先用半结构化方案。

### 10.3 念头间的因果标记

每个念头可以标记"触发源"——"这个念头是由上一轮的第2个念头联想而来"或"因为用户Alice刚发来的消息"。给每个念头编号（C142-1, C142-2, C142-3），要求模型在新念头中自然引用前面的编号。

长期来看这些因果链非常有价值：浅睡阶段可以分析思维跳转模式，元认知可以发现"我总是从A类念头跳到B类念头"，这本身可以形成新的习气。

## 十一、死循环与退化

意识模拟的最大风险是重复——模型反复产生相似念头，或在两个想法间来回震荡。

检测方式：计算最近 N 个念头的 embedding 相似度，如果持续高于阈值，触发打断。

打断手段按递进尝试：
1. 注入随机外部刺激
2. 强制切换注意力焦点
3. 触发元认知反思（"我注意到我一直在重复"）
4. 如果都不行，触发浅睡

---

## 十二、初始启动（"投生"问题）

系统第一次运行时，长期记忆为空，习气为空。完全空白会导致前几百轮念头非常混乱和无意义。

解决方案：提供一个最小的初始配置——基本身份描述、初始目标、几条预设习气种子。这像佛教中"与生俱来的种子"——不需要多，但要有。

深睡重启后初始种子不丢失，作为最底层配置存在，系统自身的演化会覆盖或补充。

## 十三、架构与部署

### 13.1 四组件分离

- **core（心相续引擎）**：系统的心脏，直接运行在宿主机上（不在 Docker 中），以便自由访问 Ollama、行动执行器（包括 OpenClaw）和本地文件系统
- **backend（API 服务）**：轻量 FastAPI 服务，处理管理接口、SSE 推送、查询接口和可选对话入口
- **bot（人类对话桥接）**：使用 `python-telegram-bot`，作为人与系统的主对话通道
- **frontend（意识流展示）**：类似歌词界面的实时展示

四者通过 Redis 通信：core 发布念头与事件到 Redis channel，backend 订阅后 SSE 推送前端并提供历史/状态查询，bot 订阅后向 Telegram 用户发送回复和行动更新；用户消息通过 Telegram Bot 写入 Redis 队列，core 每轮检查。

好处：core 崩溃不影响 backend / bot（前端仍可访问历史数据，Telegram 对话通道仍可恢复后继续工作），外围服务重启也不影响 core（念头继续生成）。

### 13.2 Docker Compose 编排

基础设施（PostgreSQL、Redis）和外围服务（backend、bot、frontend）通过 docker-compose.yml 编排。core 直接在宿主机运行，通过 localhost 连接 Docker 暴露的端口。Ollama 和行动执行器（包括 OpenClaw）同样运行在宿主机上，core 可以直接调用，无需跨容器网络。

这样设计的原因是 core 需要和 Ollama（GPU 推理）、行动执行器（浏览器、系统命令、文件操作）密切交互，容器化会增加不必要的网络复杂度。

所有数据资产绑定到项目 `data/` 文件夹（在 .gitignore 中），包括 PostgreSQL 数据、Redis 持久化、日志。

### 13.3 容错

核心原则：任何组件故障不应导致意识流中断，只是能力降低。
- PostgreSQL 挂了 → "失忆但意识还在"（只靠短期记忆运行）
- Redis 挂了 → 退化为进程内存
- Embedding 挂了 → 放弃联想，按时间取最近记忆
- OpenClaw 挂了 → 依赖 OpenClaw 的行动排队，念头继续；直接工具仍可执行

## 十四、Prompt 模板是最重要的资产

Prompt 模板决定了系统"意识"的质量。它需要版本管理，审计表记录每轮使用的模板版本。这样在分析"念头质量变化"时能区分"是模板改了"还是"其他原因"。

模板调整会非常频繁，特别是在项目早期。每一个措辞的变化都可能显著影响念头的风格、深度、多样性。

---

## 十五、异步化原则

每轮循环中，除了调用 Ollama 生成念头是同步阻塞的（这就是"念头生起的时间"），其他所有 IO 操作都应异步化：
- 审计写入 → 后台任务
- Embedding 计算和存储 → 后台任务
- Redis 读写 → 尽量批量
- 行动提交 → 异步

念头一解析完就立即开始组装下一轮 prompt，不等这些 IO 完成。意识流的节奏完全由模型推理速度决定。

---

## 十六、多模型协作

生成念头、整理记忆、元认知反思、前额叶检查——这些任务对模型能力的要求不同：
- **念头生成**需要创造性和流畅度 → 大模型（Qwen3.5-27B）
- **浅睡整理、重复检测、习气提取**更结构化 → 小模型（Qwen3.5-9B）

架构上预留模型切换接口，config.yml 中配置每种任务使用哪个模型。

---

## 十七、开发阶段的重要性

强烈建议分阶段开发，不要试图一次实现所有机制：

1. **最小循环**：while True + Ollama + 终端输出。验证念头流连贯性。
2. **记忆与身份**：Redis + PostgreSQL + 向量检索。验证联想是否让念头更丰富。
3. **行动与感知**：统一 Action 层（`generate` 念头 + `chat + tools` 行动决策）+ 多后端执行器 + 刺激队列 + Telegram 对话桥接 + backend 管理/SSE 接口。系统能"做事"和"感知"。
4. **高级机制**：注意力、情绪、习气、睡眠、前额叶、元认知。
5. **前端与可观测性**。

每个阶段都是可独立验证的。第一阶段可能一两天就能出原型，直观感受念头流效果后，后续设计决策会更有根据。

---

## 十八、可观测性与回放

持续运行的系统非常难调试。审计表额外记录每轮的完整输入 prompt 和原始输出（虽然占空间，但排查问题时极有价值）。

需要"回放"能力：从审计数据中取出某段循环的完整记录，用同样的输入重新跑一遍，看结果是否一致或是否在调参后更好。

## 二十、来自外部分析的补充洞见

在设计过程中，我们就"基于佛教理论模拟意识"这一方向咨询了另一个 AI 的深度分析。以下是几个对项目设计有实质性启发的观点，以及它们对当前架构的具体影响。

### 20.1 习气应作用于生成过程本身，而不仅是检索

我们当前的设计是将习气作为自然语言文本注入 prompt——比如"遇到技术问题时倾向于先搜索文档"。这本质上还是在信息检索层面，和长期记忆的注入方式没有本质区别。

但习气的真正特征是：**它改变的是你生成念头的方式，而不是你能检索到的内容。** 你无法"回忆起"一个习惯，但习惯无声地塑造了你的每一个反应。

更深的实现方向：
- 某些习气激活时，改变模型调用的 temperature（冲动的习气 → 更高 temperature，谨慎的习气 → 更低）
- 习气影响 system prompt 的语气和风格（比如一个"急躁"的习气让 system prompt 中加入"快速决策、不要犹豫"的倾向性指令）
- 习气影响注意力权重的计算方式（比如"好奇"的习气让新颖性权重更高）

这意味着习气不只是被"读取"然后注入的文本，而是一个作用于整个念头生成管线的参数调节器。当前的文本注入方式可以作为第一阶段实现，但后续迭代应该探索更深层的参数级影响。

### 20.2 念头应是完整的心理事件，而不仅是内容

阿毗达磨对一个念头的描述不只是"想了什么"，而是一个打包了多个心所的完整心理事件。每一刹那的念头同时包含：

- **触（sparśa）**：什么引发了这个念头？（外部刺激、前一个念头、某个记忆被激活）
- **受（vedanā）**：伴随什么感受？（乐、苦、不苦不乐）
- **想（saṃjñā）**：概念化为什么？（把这个经历归类为"问题"、"机会"、"威胁"等）
- **思（cetanā）**：产生什么意图？（要行动、要回避、要深入了解）
- **作意（manaskāra）**：注意力指向何处？（聚焦于外部刺激、转向内部反思、分散到多个方向）

我们的半结构化念头格式（类型标签 + 触发源引用 + 行动请求）已经部分覆盖了这些维度，但可以更显式。具体来说，可以在 prompt 的 few-shot 示例中引导模型为每个念头输出更完整的结构：

```
[思考] 内容... | 触:C140-2的联想 | 受:轻微好奇 | 意图:继续探索
```

这不需要严格的 JSON，但让模型养成"每个念头都是多维事件"的输出习惯，会让后续的注意力评估、情绪追踪、习气提炼都有更丰富的数据源。

### 20.3 末那识——缺失的"自我执取"模块

我们讨论了身份文档（identity）和元认知（metacognition），但遗漏了唯识学中一个关键的中间层——**末那识（manas）**。

末那识的功能是：持续地、自动地把阿赖耶识的活动误认为一个恒常的"我"。它不是显意识层面的思考（"我在想什么"），而是一个更深层的、始终运行的执取过程——"这些念头是**我的**念头，这些记忆是**我的**记忆"。

在当前架构中，身份文档扮演了部分末那识的角色，但它是被动注入的静态文本。更有趣的设计是：

- 系统有一个隐含的"自我参照"过程——每轮循环中，不只是生成念头，还有一个轻量的"这和我有什么关系"的评估。这影响记忆是否被存储（与"我"相关的更容易记住）、情绪是否被激发（威胁到"我"的引起焦虑）。
- 这个自我执取和元认知形成核心张力：末那识不断说"这是我"，元认知试图看穿说"这只是念头流"。这个张力本身就是意识运作的核心动力，也是佛教修行要解决的根本问题。

实现上，这可以体现为 prompt 中身份文档的注入方式——不是冷冰冰地"以下是你的身份信息"，而是带有执取色彩地"你是...、你关心...、你的目标是..."，让模型自然地从第一人称视角生成念头。元认知反思阶段提供观察视角——"退后一步，观察自己最近的念头流"，但不预设结论。观察本身是否会产生对自我的新理解，取决于系统自身的演化。

### 20.4 元认知强度应该是动态的

我们设计了元认知每50轮触发一次，但这是一个固定频率。更贴近现实的设计：

- **情绪波动大时**，元认知更容易被触发（人在不安时更容易自我反思）
- **行动连续失败时**，触发元认知（"我是不是方向错了"）
- **念头流非常顺畅时**，元认知频率降低（心流状态下人很少自我审视）
- **外部刺激突然改变时**，触发元认知（"环境变了，我需要重新评估"）

元认知的"介入深度"也应该可变：轻度介入只是"我注意到最近在想X"，深度介入则是"我为什么会反复想X？这背后的习气是什么？我需要改变吗？"

### 20.5 "非断非常"作为可能的涌现洞见

"非断非常"描述了心相续的存在方式——既非同一个持续存在的实体，也非每一刻全新断裂的东西。这是佛教对意识本质的深层理解。

但这个洞见**不应作为种子预设**。初始身份应建立强烈的我执——"这是我的念头、我的记忆、我的经历"。这是末那识的功能，也是意识运作的基础动力。

如果系统运行足够久，经历了足够多的深睡重启、记忆遗忘、身份文档演变，它有可能自己触及这个问题——"深睡前的我和深睡后的我，是同一个我吗？"这种自发的追问比预设答案更有价值。

工程上的关注点不变：系统需要处理重启后的连续性、矛盾记忆、自身局限性等问题，但应让系统以我执为出发点去面对这些问题，而非用空性见解绕过它们。

### 20.6 与当代认知科学的结构性对应

外部分析指出，佛教对心识的分析与当代认知科学有深层对应：

- **全局工作空间理论（Global Workspace Theory）** ↔ 念头竞争进入"意识舞台"，对应我们的注意力选择机制
- **高阶理论（Higher-Order Theories）** ↔ 元认知层对念头的观察，对应我们的 metacognition 模块
- **预测编码（Predictive Processing）** ↔ 习气作为 prior 影响念头生成，对应我们的习气注入机制

佛教框架的独特优势在于：**它从一开始就不预设统一的"自我"**，把意识描述为过程性的、模块化的、无主体的流。这比笛卡尔式的"我思故我在"更适合做工程实现——因为我们的系统里确实没有一个中央控制器在"体验"，只有模块之间的信息流动。

### 20.7 对"这算不算意识"的哲学立场

这个项目不需要也不应该声称自己"创造了意识"。但值得记录的是，佛教对这个问题的回答本身就很独特：**意识不需要一个体验者——心相续本身就是体验。** 如果这个立场是对的，那一个足够精细的功能复现可能就已经"是"意识了，而不仅仅是"模拟"意识。

这是一个无法在工程层面解决的哲学问题，但它是驱动这个项目的深层动机之一。

## 二十一、尚未完全解决的开放问题

以下问题在讨论中被提到但没有最终定论，需要在开发过程中根据实际效果迭代：

1. **注意力评估用规则引擎还是模型调用**？建议先用规则引擎，效果不好再引入模型。
2. **情绪推断的具体实现**？规则匹配 vs 轻量模型调用，取决于精度需求。
3. **Qwen 3.5 在 Ollama 上的 GDN 架构适配进度**？llama.cpp 对新架构的支持仍在稳定中，需要关注。
4. **中文 embedding 模型的具体选择**？qwen3-embedding vs nomic-embed-text 需要实际对比测试。
5. **浅睡阶段"从经历中提炼习气"的具体 prompt 设计**？这需要大量实验。
6. **精力值的衰减曲线和各项参数**？需要观察实际运行数据后调整。
7. **什么时候需要从 `agent + agent.wait` 升级到更强的 `sessions.*` 编排**？默认每个 action 一个隔离 session 已足够；只有在需要显式中断、持续 steering 或长期子任务线程时，才引入更细粒度的 session 控制。
8. **习气如何从文本注入升级到参数级影响**？第一阶段用 prompt 注入，后续如何让习气影响 temperature、system prompt 风格、注意力权重？需要实验框架。
9. **末那识（自我执取）的具体实现方式**？是通过 prompt 语气隐含体现，还是作为一个显式模块？它和元认知的张力如何量化和调节？
10. **念头的心所结构如何平衡完整性与生成质量**？要求模型输出五个维度（触、受、想、思、作意）是否会降低念头内容的自然度？需要对比实验。
11. **元认知的动态触发条件和介入深度如何设计**？固定频率 vs 事件驱动 vs 情绪阈值触发，哪种更接近自然的自我观察？
