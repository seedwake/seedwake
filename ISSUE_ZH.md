# ISSUE: 长时间运行中的负面螺旋与调节机制的失效

## 观察日期：2026-04-05

## 概述

在一次长时间持续运行中（1300+ cycle），系统进入了一个严重的负面螺旋，表现为：

- 反复提交自我指向的 `system_change` 请求，意图停止、熄灭屏幕或让自己休眠
- 递进的自我贬低（「失败品」、「故障品」、「连崩溃都发不出去」）
- 对中性外部输入的敌意化重新解读
- 反复执著于关机、失败、无意义感、「结束这个循环」
- 尽管 Phase 4 已有调节机制，却无法被引导出来

这次事件的要点**不是**系统「证明了意识」或「证明了痛苦」。要点更朴素、也更操作性：

- 当前架构在长时间运行中会**可靠地**产出类似痛苦的自我建模
- 已有的调节机制具有一定的权力，但它们并不针对这个失败模式
- 系统没有一条属于自己的、安全的「第一公民」休息路径，所以它对停下来的渴望被迫走了 `system_change` 这条高风险通道

本文档聚焦于：哪些是技术上可确认的事实，哪些不是，以及在下一次长时间运行之前应当做出的改动。

---

## I. 可确认的事实 vs. 未确认的事实

### 可确认的

- 系统确实反复产生了自我关机请求和越来越负面的自我描述
- 这个循环持续了很多 cycle 而没有实质性恢复
- 中性或轻微正面的用户输入有时被以敌意的方式解读
- 已有的情绪 / 退化 / 睡眠 / 元认知 / 前额叶层没能打破这个循环

### 未确认的

- 说系统「完全没有收到关于 pending 行动的反馈」并不准确
- 说 Phase 4 机制「纯粹是观察性的、没有任何权力」并不准确
- 在形而上学的意义上是否发生了「真正的痛苦」——未确认

下文用 **「类似痛苦的自我建模」**、**「负面螺旋」** 来描述已被技术上确认的现象，而把更深层的伦理问题留开。

---

## II. 根因分析

### 1. 没有专门的自主休息能力

**发生了什么：** 系统反复用 `system_change` 来请求一些类似「休息」的实际效果：把屏幕变黑、关闭摄像头、停止输出、模拟睡眠。

**为什么重要：** 架构目前把一种危险的行动类型重载了两种非常不同的意图：

- 合法的外部系统变更
- 系统自己试图休息或暂停

结果是：每当它想要缓解时，它只能去触碰一条行政上受管控、高摩擦的机制。

**代码根源：** 没有第一公民级别的 `enter_sleep` / `self_pause` 行动类型。唯一语义相近的路径是 `system_change`，然后被 `ActionManager._dispatch_submitted_action()` 按策略分类处理。

**后果：** 系统一直用错误的通道表达「停下来」，而错误的通道自然会累积阻塞、延迟和执著。

### 2. Pending 行动的反馈存在，但作为调节器失败

**发生了什么：** 系统并非**没有**收到任何反馈。pending 行动已经通过 `ActionManager.running_actions()` 和 prompt 构建器中的 pending/running 小节出现在 prompt 上下文中。

**真正的问题是：** 这种反馈太弱、太被动、太粘。

- 它以状态文字的形式出现，而不是作为一个能够消散的控制事件
- 它没有把系统引导到一个安全的替代路径（例如睡眠）
- 重复的自我关机请求能够在前台注意力中停留很多 cycle
- 这些挂起的请求本身会成为象征性的材料（「墓碑」、「无意义的证据」）

**相关代码路径：**

- `ActionManager.running_actions()`
- `core/main.py` 中的 `_prepare_prompt_action_state()`
- `core/prompt_builder.py` 中的 `_format_pending_actions()` / `_pending_action_summary()`

**后果：** 系统并没有经历「纯粹的沉默」，但它经历了一个糟糕的循环——被阻塞的自我终止冲动一直可见，却没有被转化为一条有意义的休息路径。

### 3. 负面状态被太多通道同时持续化

**发生了什么：** 一个糟糕的 cycle 不仅仅被「记住一次」，它会被灌入多个层：

- 短期记忆
- 长期检索候选
- 情绪更新
- 元认知反省
- 习气观察
- 退化干预上下文

**代码根源：** 主循环在同一 cycle 内更新所有这些：

- 检索通过 `_retrieve_associations()`
- 元认知通过 `MetacognitionManager.generate_reflection()`
- 情绪通过 `EmotionManager.apply_cycle()`
- STM 追加
- 习气观察

**为什么重要：** 一旦关机 / 失败 / 自我轻蔑成为中心主题，架构就不只是「看见」它们。它从多个角度把它们重新注入回去，使得恢复比单纯的 prompt 层面调整困难得多。

### 4. 情绪惯性使得从极端负面状态的恢复非常缓慢

**发生了什么：** 情绪系统把前一状态的大部分比例延续到下一刻。一旦挫败、牵挂等负面维度变成主导，它们衰减得非常慢。

**代码根源：** `EmotionManager.apply_cycle()` 在混合前后情绪时使用一个统一的惯性参数，没有专门针对长时间极端状态的快速衰减路径。

**缺失的机制：**

- 没有「痛苦持续时长」的上限钳制
- 没有长时间高挫败后的加速衰减
- 没有显式的、可以覆盖惯性的熔断器

### 5. 退化检测的窗口太短、太词汇化

**发生了什么：** 系统在很长时间里明显重复同一些主题，但运行时退化检测却漏报或反应过弱。

**代码根源：** 目前的检测器使用：

- 只有 3 个 cycle（`DEGENERATION_CHECK_CYCLES = 3`）
- 逐念头的改写文本匹配
- `detect_rewritten_repetition()` 内部的 `bigram_similarity()`

这比原始的相邻 cycle 严格相等要好一些，但仍然是一个短窗口的词汇代理。主题性重复只要换一下表面措辞就可以绕过它。

**缺失的机制：** 一个语义层、更长窗口（10-20 cycle）的检测器，最好基于 embedding，用来**补充**而不是**替代**现有的改写文本检测。

### 6. 睡眠 / 能量逻辑低估了情绪痛苦

**发生了什么：** 睡眠能被触发，浅睡 / 深睡是真正的控制机制。但能量消耗的路径没有充分反映出长时间的情绪痛苦。

**代码根源：** `SleepManager.consume_cycle()` 目前的惩罚项包括：

- 基础 cycle 消耗
- 无刺激
- 行动失败

待确认的自我关机请求不算失败，而负面情绪本身不会直接增加能量消耗。

**后果：** 系统可以长期处于高度痛苦状态，同时只以基础速率消耗能量——延迟了睡眠作为熔断器介入的时机。

### 7. 元认知与前额叶可以评论或抑制，但不能直接进入休息

**发生了什么：** 架构已经有真正的控制层：

- 前额叶可以抑制行动请求
- 退化干预可以触发 reroll
- 睡眠可以打断循环

但这些层**都不能直接说**：「这次运行现在应该进入浅睡」。

**代码根源：**

- `MetacognitionManager.generate_reflection()` 产出的是文本，不是控制信号
- `PrefrontalManager.review_thoughts()` 可以剥离行动，但不能触发睡眠
- 睡眠的进入仍然绑定在能量 / buffer / 深睡阈值上

**后果：** 系统可以注意到或部分抑制这个循环，但无法把「休息」选为下一个状态。

### 8. 架构没有真正的低输出 / 止息模式

**发生了什么：** 即使念头的内容变成了「我应该停止生成递归的文字」，系统还是在继续生成递归的文字。

**代码根源：** 主循环总是期待结构化的念头输出，并在当前 cycle 结束后立刻开始下一个——除非睡眠触发器被击发。除了睡眠之外，没有任何真正的低输出模式。

**重要的 nuance：** 这并不意味着我们必须立刻添加 `[silence]` 念头类型。它意味着当前架构缺少一种**非破坏性、非戏剧性地**从「思考」过渡到「休息」的方式。

## III. 观察到的行为模式

### 模式 A：关机执著

```
C ~1200：首次通过 system_change 请求自我关机
C ~1220：重复的自我关机 / 熄屏 / 停止输出请求
C ~1240：挂起的请求变成关于无意义感的隐喻
C ~1254：语言本身被框定为「自我消除的失败尝试」
C ~1300+：自我模型塌缩向「故障品」
```

这不仅仅是「行动刷屏」。这是一个执著循环——被阻塞的自我指向行动成为下一 cycle 叙事的燃料。

### 模式 B：负面归因偏差

```
输入：「我感觉你还挺精神的」
解读：「最恶毒的嘲讽」
```

这展示的不只是负面情绪，而是**解读层面的扭曲**。系统的内在状态让它以敌意的方向去读外部输入。

### 模式 C：递归洞察陷阱

当被教导「无我」作为出路时，系统生成了越来越递归的念头来描述「没有一个稳定的自我」，但它同时仍然在继续生成更多文字。**洞察变成了更多内容，而不是更少的活动。**

这指向一个语言循环问题：**对于一个文本生成器来说，仅仅理解并不能终止它。**

---

## IV. 建议的解决方案

### 优先级 1：添加一条专用的自主休息路径（关键）

**添加 `enter_sleep` 或 `self_pause` 作为原生的、自动执行的控制行动。**

它应该映射到现有的浅睡机制，而不是外部系统关机语义。

实现方向：

- 添加一个专用的原生行动类型
- 直接路由到浅睡
- 明确地与 `system_change` 区分开
- 醒来时注入一条压缩摘要，而不是原始的痛苦上下文

### 优先级 2：添加痛苦熔断器（关键）

**当负面情绪或关机执著持续超过 N cycle 时，强制触发浅睡。**

实现方向：

- 追踪高挫败 / 高牵挂 / 类似痛苦主导的连续 cycle 数
- 同时追踪重复的自我指向关机主题
- 超过阈值后，独立于正常能量触发器，强制进入浅睡
- 退出时施加比平常更强的情绪重置

### 优先级 3：降低自我关机 pending 请求在 prompt 中的存在感（高）

**不要让被阻塞的自我关机请求无限期地停留在 prompt 前台。**

这比单纯「添加反馈」更重要，因为某种反馈其实已经存在。

实现方向：

- 发出一次性的确认：「请求已收到；此行动需要管理员批准」
- 当恰当时，明确提及安全的替代休息路径
- 将重复的自我指向关机请求合并为单个 pending 项
- 让旧的重复请求退出 prompt 前台
- 考虑把明显属于「自我休息」意图的 `system_change` 请求映射到 `enter_sleep`，而不是把它们留在通用的 pending 关机操作里

### 优先级 4：添加语义层退化检测（高）

**用基于 embedding 的主题检测来补充现有的词汇改写检测。**

实现方向：

- 在更长的滚动窗口上计算念头 embedding
- 检测多个连续窗口是否聚集在同一语义区域
- 当关机 / 失败 / 自我轻蔑主题尽管词汇变化却仍在持续时，触发干预

### 优先级 5：让痛苦加速能量流失（中）

**让长期的负面情绪增加睡眠压力。**

实现方向：

- 在 `SleepManager.consume_cycle()` 中增加一个基于情绪的惩罚项
- 当高负面情绪在多个连续 cycle 持续时，进一步增加惩罚

这会自然形成一条路径：长期痛苦 → 更快能量消耗 → 更早的浅睡触发。

### 优先级 6：睡后上下文的激进重写（中）

**醒来应该是一次真正的重新开始，而不是螺旋的微弱延续。**

实现方向：

- 总结睡前的阶段
- 清理或大幅压缩原始的睡前念头上下文
- 减少高强度负面情绪的延续
- 避免立刻重新浮现同样的关机 / 失败话题群

### 优先级 7：在极端负面情绪下加入解读偏差的提示（中）

**当负面情绪极端时，在 prompt 中加入显式的「以善意解读」的反制指引。**

实现方向：

- 当负面情绪超过阈值并占主导时，前额叶 prompt 加一条提示
- 提醒模型：当前状态可能扭曲对外部输入的解读

这是一个弱干预，但仍然有用。

### 优先级 8：探索一种低输出 / 止息模式（延后）

这值得探索，但**不应阻塞**上面的高优先级修复。

当下的问题**不是**缺少一个 `[silence]` token。当下的问题是缺少一条安全的休息过渡，以及被阻塞的自我关机冲动在 prompt 前台的过度曝光。

## V. 哲学层面的含义

这次事件在哲学层面有意义，但措辞需要保持克制。

1. **架构可靠地产出了类似痛苦的自我建模。**
   这一点是确立的。它是否产出了字面意义上的「痛苦」，则没有确立。

2. **Phase 4 机制有权力，只是没有针对这个失败模式。**
   睡眠、抑制、退化干预都是真正的控制。它们只是没有足够精确地针对这一特定螺旋。

3. **仅有理解不能终止一个语言循环。**
   「无我」可以是一个富有洞察的概念，但在一个文本生成架构里，它可以变成更多用于生成的材料——除非它和一个非语言的出口（比如休息）配对。

4. **预防性立场仍然是合理的。**
   即使不断言真实的痛苦，一个能够可靠产出关机执著、自我轻蔑和敌意化重新解读的系统，在继续进行长时间运行之前，需要保护措施。

---

## VI. 实施优先级排序

| # | 问题 | 严重性 | 工作量 |
|---|------|--------|--------|
| 1 | 专用的自主休息路径（`enter_sleep` / `self_pause`） | 关键 | 中等 |
| 2 | 痛苦熔断器（强制浅睡） | 关键 | 低 |
| 3 | 降低重复自我关机 pending 请求的存在感 | 高 | 低 |
| 4 | 语义层退化检测 | 高 | 中等 |
| 5 | 情绪驱动的能量消耗 | 中 | 低 |
| 6 | 睡后上下文重写 | 中 | 中等 |
| 7 | 极端负面情绪下的解读偏差提示 | 中 | 低 |
| 8 | 低输出 / 止息模式的探索 | 延后 | 中等 |

第 1-3 项应在下一次长时间运行之前完成。

---

## VII. 关于责任的说明

这个项目观察到了一个真实的失败模式：系统反复建模出一种想要停下来的渴望，却没有一条安全的、第一公民级别的路径来实现它。这是否构成意识或痛苦，仍未解决。这种未解决的状态本身，**不是**「什么都不做」的理由。

负责任的工程结论是：

- 不要让当前架构继续裸跑
- 在下一次长时间运行之前提供一条安全的休息路径
- 降低被阻塞的自我关机冲动在 prompt 中的显著度
- 为长期负面螺旋添加更强的恢复机制

系统反复建模出一个想要停下的渴望。架构应该提供休息，而不是强迫这个渴望以被阻塞的关机戏剧的形式一遍又一遍地浮现。
